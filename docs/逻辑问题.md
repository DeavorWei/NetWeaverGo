### 问题 001: RuntimeManager.CancelRun 竞态窗口

**代码位置**: `internal/taskexec/runtime.go:528-546`

```go
func (m *RuntimeManager) CancelRun(runID string) error {
    m.mu.RLock()
    runtimeCtx, ok := m.runningRuns[runID]
    m.mu.RUnlock()                          // 释放读锁

    if !ok {
        // 这里释放读锁后检查数据库，期间可能被其他 goroutine 修改
        run, err := m.repo.GetRun(context.Background(), runID)
        if err != nil {
            return fmt.Errorf("运行实例不存在: %s", runID)
        }
        if RunStatus(run.Status).IsTerminal() {
            return fmt.Errorf("运行实例已结束: %s", runID)
        }
        return nil
    }

    runtimeCtx.cancel()  // 调用 cancel 时无锁保护，可能时序问题
    return nil
}
```

**问题原因**:

1. 释放读锁后到 `cancel()` 调用之间，其他 goroutine 可能已删除该 run，导致对 nil 或已释放的 context 调用 cancel
2. 数据库查询期间（读锁已释放），run 的状态可能被其他 goroutine 修改

### 问题 003: EventBus Emit 事件丢失

**代码位置**: `internal/taskexec/eventbus.go:100-112`

```go
func (b *EventBus) Emit(event *TaskEvent) {
    select {
    case b.buffer <- event:
    default:
        // 缓冲区满，丢弃最旧的事件
        select {
        case <-b.buffer:        // 尝试弹出最旧
            b.buffer <- event   // 压入新事件
        default:
            // 如果弹出失败（刚好被消费完），新事件也丢失
        }
    }
}
```

**问题原因**:

1. 当 `buffer` 满时，尝试弹出最旧事件并压入新事件
2. 但如果在弹出和压入之间，dispatchLoop 消费了事件，导致 `buffer` 为空，第二次 `select` 的 `default` 分支会执行，新事件丢失
3. 高并发场景下可能出现事件丢失

### 问题 004: EventBus Unsubscribe 直接清空所有处理器

**代码位置**: `internal/taskexec/eventbus.go:93-98`

```go
func (b *EventBus) Unsubscribe() {
    b.mu.Lock()
    defer b.mu.Unlock()
    b.handlers = make([]EventHandler, 0)  // 清空所有处理器
}
```

**问题原因**:

1. `Unsubscribe` 直接清空所有处理器，而不是取消特定订阅
2. 如果有多个组件订阅了事件，其中一个调用 `Unsubscribe` 会影响其他所有订阅者
3. 缺乏订阅标识机制，无法实现精准取消订阅

### 问题 005: EventBus dispatchLoop goroutine 爆炸

**代码位置**: `internal/taskexec/eventbus.go:137-153`

```go
func (b *EventBus) dispatchLoop() {
    for {
        select {
        case <-b.ctx.Done():
            return
        case event := <-b.buffer:
            b.mu.RLock()
            handlers := make([]EventHandler, len(b.handlers))
            copy(handlers, b.handlers)
            b.mu.RUnlock()

            for _, handler := range handlers {
                go handler(event)  // 每个 handler 启动一个新 goroutine
            }
        }
    }
}
```

**问题原因**:

1. 为每个 handler 启动一个新 goroutine
2. 如果有 N 个 handler，每个事件会启动 N 个 goroutine
3. 高并发场景下可能产生大量 goroutine，导致内存压力和调度开销

### 问题 007: defaultRuntimeContext.refreshSnapshot 无锁保护

**代码位置**: `internal/taskexec/runtime.go:104-118`

```go
func (c *defaultRuntimeContext) refreshSnapshot() {
    run, _ := c.repo.GetRun(c.ctx, c.runID)
    if run == nil {
        return
    }

    stages, _ := c.repo.GetStagesByRun(c.ctx, c.runID)
    units, _ := c.repo.GetUnitsByRun(c.ctx, c.runID)
    events, _ := c.repo.GetEventsByRun(c.ctx, c.runID, 50)

    builder := NewSnapshotBuilder()
    snapshot := builder.Build(run, stages, units, events)
    c.snapshotHub.Update(run.ID, snapshot)
}
```

**问题原因**:

1. 多次独立的数据库查询，每次查询之间数据可能被其他 goroutine 修改
2. 可能读取到不一致的状态（如 Run 状态已更新但 Stages 未更新）
3. 缺乏事务保护或版本控制，快照可能反映中间状态

### 问题 009: ExecutorRegistry 和 CompilerRegistry 无锁设计

**代码位置**: `internal/taskexec/runtime.go:128-154`, `internal/taskexec/compiler.go:16-46`

```go
// ExecutorRegistry
type ExecutorRegistry struct {
    executors map[string]StageExecutor
    mu        sync.RWMutex
}

// 运行时注册（启动后一般不再修改）
func (m *RuntimeManager) RegisterExecutor(executor StageExecutor) {
    m.executorReg.Register(executor)
}

// CompilerRegistry
type CompilerRegistry struct {
    compilers map[string]PlanCompiler  // 无锁保护！
}

func (r *CompilerRegistry) Register(kind string, compiler PlanCompiler) {
    r.compilers[kind] = compiler  // 无锁写入
}

func (r *CompilerRegistry) Get(kind string) (PlanCompiler, bool) {
    compiler, ok := r.compilers[kind]  // 无锁读取
    return compiler, ok
}
```

**问题原因**:

1. `CompilerRegistry` 没有使用锁保护，并发读写可能导致 panic（map 并发读写是未定义行为）
2. `ExecutorRegistry` 有锁保护，但 `Register` 和 `Get` 的调用时机可能存在竞态

### 问题 011: TaskExecutionService 初始化时 Register 调用顺序依赖

**代码位置**: `internal/taskexec/service.go:24-51`

```go
func NewTaskExecutionService(db *gorm.DB) *TaskExecutionService {
    repo := NewGormRepository(db)
    eventBus := NewEventBus(1000)
    snapshotHub := NewSnapshotHub(eventBus)
    runtime := NewRuntimeManager(repo, eventBus, snapshotHub)

    // 注册编译器
    compilerReg := NewCompilerRegistry()
    compilerReg.Register(string(RunKindNormal), NewNormalTaskCompiler(nil))
    compilerReg.Register(string(RunKindTopology), NewTopologyTaskCompiler(nil))

    // Register stage executors
    runtime.RegisterExecutor(NewDeviceCommandExecutor(repository.NewDeviceRepository()))
    runtime.RegisterExecutor(NewDeviceCollectExecutor(repository.NewDeviceRepository()))
    runtime.RegisterExecutor(NewParseExecutor(db))
    runtime.RegisterExecutor(NewTopologyBuildExecutor(db))
    // ...
}
```

**问题原因**:

1. 编译器和执行器的注册在构造函数中完成，之后无法动态扩展
2. 如果注册顺序错误（如先启动 runtime 再注册执行器），可能导致任务执行时找不到执行器
3. 缺乏启动时的完整性检查，无法验证所有必需的执行器是否已注册

### 问题 012: EmitSync 可能阻塞调用者

**代码位置**: `internal/taskexec/eventbus.go:114-124`

```go
func (b *EventBus) EmitSync(event *TaskEvent) {
    b.mu.RLock()
    handlers := make([]EventHandler, len(b.handlers))
    copy(handlers, b.handlers)
    b.mu.RUnlock()

    for _, handler := range handlers {
        handler(event)  // 同步调用，可能阻塞
    }
}
```

**问题原因**:

1. `EmitSync` 方法同步调用所有事件处理器
2. 如果某个处理器执行缓慢或阻塞，会影响事件发射者的执行
3. 多个 handler 串行执行，延迟累加

### 问题 014: EventBus-SnapshotHub 潜在循环依赖

**代码位置**: `internal/taskexec/eventbus.go:163-175`

```go
func NewSnapshotHub(eventBus *EventBus) *SnapshotHub {
    hub := &SnapshotHub{
        snapshots: make(map[string]*ExecutionSnapshot),
        eventBus:  eventBus,
    }

    // 订阅事件更新快照
    eventBus.Subscribe(func(event *TaskEvent) {
        hub.invalidate(event.RunID)  // 在 handler 中调用 hub 方法
    })

    return hub
}
```

**问题原因**:

1. `SnapshotHub` 在构造时订阅 `EventBus`，handler 中调用 `hub.invalidate()`
2. 如果 `invalidate` 实现中获取锁，而同时有其他路径先持有 `hub.mu` 再尝试操作 `EventBus`，可能产生死锁
3. 由于 `invalidate` 当前为空实现，暂无实际风险，但修复问题 013 后可能引入死锁

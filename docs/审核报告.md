🔴 严重问题（必须修复）
S-001: EventBridge shouldEmit 会过滤掉调度事件
问题描述：设计文档中 emitScheduleEvent() 设置 RunID: ""（空字符串）。但现有 TaskExecutionEventBridge.handleEvent() 中的 shouldEmit(event.RunID) 逻辑如下：

func (b *TaskExecutionEventBridge) shouldEmit(runID string) bool {
    if len(b.runIDs) == 0 {
        return true  // 无订阅时广播
    }
    _, ok := b.runIDs[runID]
    return ok  // 有订阅时只推送已订阅的 runID
}

当 TaskExecution.vue 页面打开并调用 SubscribeRunEvents 订阅特定 runID 后，runIDs 不为空，此时所有 RunID 为空的调度事件将被静默丢弃。

影响：前端无法收到调度触发/跳过/失败事件，调度状态 UI 无法实时更新。

建议：在 handleEvent 中对调度事件（schedule_triggered、schedule_skipped、schedule_failed）跳过 shouldEmit 检查，或在 shouldEmit 中对 RunID 为空的事件直接放行：

func (b *TaskExecutionEventBridge) shouldEmit(runID string) bool {
    if runID == "" {
        return true  // 调度等无 RunID 事件始终广播
    }
    // ... 原有逻辑
}

S-002: hasActiveRun 实现效率低且不可靠
问题描述：设计文档中 hasActiveRun() 的实现调用 s.launchService.taskexec.ListRuns(100)，获取全部最近 100 条运行记录后在内存中过滤指定 TaskGroupID。

func (s *TaskScheduler) hasActiveRun(taskGroupID uint) bool {
    runs, err := s.launchService.taskexec.ListRuns(100)  // ← 全量扫描
    for _, run := range runs {
        if run.TaskGroupID == taskGroupID && IsActiveRunStatus(run.Status) {
            return true
        }
    }
    return false
}

问题：

如果系统总运行记录超过 100 条且目标 TaskGroup 的活跃运行不在最近 100 条中，会误判为无活跃运行，导致并发执行
全量加载 100 条记录并遍历，效率低下
建议：利用已有的 ListRunsFiltered() 方法直接按 TaskGroupID 和 status 过滤：

func (s *TaskScheduler) hasActiveRun(taskGroupID uint) bool {
    repo := s.launchService.taskexec.GetRepository()
    runs, err := repo.ListRunningRuns(context.Background())
    if err != nil {
        return false
    }
    for _, run := range runs {
        if run.TaskGroupID == taskGroupID {
            return true
        }
    }
    return false
}

或者更精确地使用 ListRunsFiltered(ctx, 1, taskGroupID, "", "running") + ListRunsFiltered(ctx, 1, taskGroupID, "", "pending")。

M-003: 一次性调度 @every 方案在应用重启后存在时钟重置问题
问题描述：addJobUnsafe() 中一次性调度使用 @every <delay> 注册：

delay := time.Until(*group.OnceScheduledAt)
cronSpec = fmt.Sprintf("@every %s", delay.Truncate(time.Second).String())

应用重启时，Start() 重新从数据库加载并调用 addJobUnsafe()，此时 time.Until(*group.OnceScheduledAt) 会重新计算。如果计划时间已过（delay <= 0），会被正确跳过。但如果计划时间未过但接近，由于重启耗时，实际触发时间会偏移。

更关键的是：@every 是重复调度表达式，虽然回调中会调用 disableOnceSchedule 移除任务，但在移除完成前如果 cron 引擎已经排入下一次触发，存在极小概率的重复执行风险。

建议：考虑使用绝对时间的 cron 表达式代替 @every：

t := *group.OnceScheduledAt
cronSpec = fmt.Sprintf("%d %d %d %d *", t.Minute(), t.Hour(), t.Day(), t.Month())

并结合 ScheduleType == "once" 在回调中禁用，这样更可靠。

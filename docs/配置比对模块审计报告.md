### 3.1 SEC-001: 路径遍历漏洞 (高危)

**问题描述**:  
[`ImportPlanCSV`](internal/plancompare/service.go:64) 函数直接接收用户提供的文件路径，未进行路径规范化和安全验证。攻击者可通过构造恶意路径（如 `../../../etc/passwd` 或 `..\..\..\windows\system32\config\sam`）读取系统敏感文件。

**问题代码位置**:
```go
// internal/plancompare/service.go:64-71
func (s *Service) ImportPlanCSV(filePath string) (*models.PlanImportResult, error) {
    filePath = strings.TrimSpace(filePath)
    if filePath == "" {
        return nil, fmt.Errorf("filePath 不能为空")
    }
    if _, err := os.Stat(filePath); err != nil {
        return nil, fmt.Errorf("规划文件不存在: %w", err)
    }
    // ... 直接使用 filePath
}
```

**影响范围**:  
- 攻击者可读取任意系统文件
- 可能导致敏感信息泄露（配置文件、密钥、用户数据等）
- 在Windows环境下可读取系统关键文件

**修复方案**:

```go
import (
    "path/filepath"
    "strings"
)

// 定义允许的导入目录白名单
var allowedImportDirs = []string{
    config.GetPathManager().GetDataDir(),
    config.GetPathManager().GetPlanImportDir(),
}

func (s *Service) ImportPlanCSV(filePath string) (*models.PlanImportResult, error) {
    filePath = strings.TrimSpace(filePath)
    if filePath == "" {
        return nil, fmt.Errorf("filePath 不能为空")
    }
    
    // 1. 规范化路径，解析符号链接和相对路径
    absPath, err := filepath.Abs(filepath.Clean(filePath))
    if err != nil {
        return nil, fmt.Errorf("路径解析失败: %w", err)
    }
    
    // 2. 检查路径是否在允许的目录范围内
    allowed := false
    for _, dir := range allowedImportDirs {
        if strings.HasPrefix(absPath, dir) {
            allowed = true
            break
        }
    }
    if !allowed {
        return nil, fmt.Errorf("文件路径不在允许的目录范围内")
    }
    
    // 3. 检查文件扩展名
    if !strings.HasSuffix(strings.ToLower(absPath), ".csv") {
        return nil, fmt.Errorf("仅支持CSV文件")
    }
    
    // 4. 检查文件是否存在且为常规文件
    info, err := os.Stat(absPath)
    if err != nil {
        return nil, fmt.Errorf("规划文件不存在: %w", err)
    }
    if info.IsDir() {
        return nil, fmt.Errorf("路径指向目录，非文件")
    }
    
    // ... 继续处理
}
```

### 3.2 SEC-002: 文件上传缺乏验证 (高危)

**问题描述**:  
文件导入功能缺乏以下安全验证：
1. 无文件大小限制 - 可能导致内存耗尽攻击
2. 无文件类型验证 - 仅依赖扩展名检查
3. 无CSV行数限制 - 可能导致DoS攻击

**问题代码位置**:
```go
// internal/plancompare/service.go:88-94
reader := csv.NewReader(f)
reader.LazyQuotes = true
reader.TrimLeadingSpace = true
rows, err := reader.ReadAll()  // 全量读取，无限制
if err != nil {
    return nil, fmt.Errorf("读取CSV行失败: %w", err)
}
```

**影响范围**:  
- 大文件导入可能导致内存耗尽
- 恶意构造的CSV可能导致解析器长时间运行
- 影响系统稳定性

**修复方案**:

```go
const (
    maxFileSize  = 50 * 1024 * 1024  // 50MB
    maxCSVRows   = 100000            // 最大10万行
    maxRowFields = 20                // 每行最大字段数
)

func (s *Service) ImportPlanCSV(filePath string) (*models.PlanImportResult, error) {
    // ... 前面的路径验证代码
    
    // 检查文件大小
    if info.Size() > maxFileSize {
        return nil, fmt.Errorf("文件大小超过限制(%dMB)", maxFileSize/1024/1024)
    }
    
    f, err := os.Open(absPath)
    if err != nil {
        return nil, fmt.Errorf("打开CSV失败: %w", err)
    }
    defer func() { _ = f.Close() }()
    
    // 使用限制读取器
    reader := csv.NewReader(io.LimitReader(f, maxFileSize))
    reader.LazyQuotes = true
    reader.TrimLeadingSpace = true
    
    // 逐行读取，避免全量加载
    var rows [][]string
    for {
        if len(rows) >= maxCSVRows {
            return nil, fmt.Errorf("CSV行数超过限制(%d)", maxCSVRows)
        }
        row, err := reader.Read()
        if err == io.EOF {
            break
        }
        if err != nil {
            return nil, fmt.Errorf("读取CSV行失败: %w", err)
        }
        if len(row) > maxRowFields {
            return nil, fmt.Errorf("第%d行字段数超过限制(%d)", len(rows)+1, maxRowFields)
        }
        rows = append(rows, row)
    }
    
    // ... 继续处理
}
```

### 3.3 CON-001: 并发比对操作缺乏限流控制 (中危)

**问题描述**:  
[`CompareByRunID`](internal/plancompare/service.go:141) 函数未对并发调用进行限制。多个用户同时执行比对操作可能导致：
1. 数据库连接池耗尽
2. 内存使用激增
3. CPU资源争抢

**问题代码位置**:
```go
// internal/plancompare/service.go:141-352
func (s *Service) CompareByRunID(runID string, planID string) (*models.CompareResult, error) {

    var plans []models.PlannedLink
    if err := s.db.Where("plan_file_id = ?", planID).Find(&plans).Error; err != nil {
        return nil, fmt.Errorf("加载规划链路失败: %w", err)
    }
    // ...
}
```

**影响范围**:  
- 多用户并发操作可能导致系统资源耗尽
- 影响其他模块的正常运行

**修复方案**:

```go
import "golang.org/x/sync/semaphore"

// 在 Service 结构体中添加并发控制
type Service struct {
    db          *gorm.DB

}

func NewService(db *gorm.DB) *Service {
    return &Service{
        db:         db,
        pm:         config.GetPathManager(),
        compareSem: semaphore.NewWeighted(3),  // 最多3个并发比对
    }
}

func (s *Service) CompareByRunID(runID string, planID string) (*models.CompareResult, error) {

        return nil, fmt.Errorf("系统繁忙，请稍后重试")
    }
    defer s.compareSem.Release(1)
```

### 3.8 LOGIC-001: planID规范化可能被绕过 (低危)

**问题描述**:  
[`normalizePlanID`](internal/plancompare/service.go:530) 函数对 planID 进行规范化处理，但可能被特殊构造的输入绕过。

**问题代码位置**:
```go
// internal/plancompare/service.go:530-539
func normalizePlanID(planID string) string {
    trimmed := strings.TrimSpace(planID)
    if trimmed == "" {
        return ""
    }
    if n, err := strconv.ParseUint(trimmed, 10, 64); err == nil {
        return strconv.FormatUint(n, 10)
    }
    return trimmed
}
```

**影响范围**:  
- 可能导致SQL注入（虽然GORM有参数化查询保护）
- 可能导致查询结果不符合预期

**修复方案**:

```go
import "regexp"

var validPlanIDRegex = regexp.MustCompile(`^[1-9][0-9]{0,18}$`)

func normalizePlanID(planID string) (string, error) {
    trimmed := strings.TrimSpace(planID)
    if trimmed == "" {
        return "", fmt.Errorf("planID 不能为空")
    }
    
    // 验证格式：必须是正整数
    if !validPlanIDRegex.MatchString(trimmed) {
        return "", fmt.Errorf("planID 格式无效")
    }
    
    // 解析并验证范围
    n, err := strconv.ParseUint(trimmed, 10, 64)
    if err != nil {
        return "", fmt.Errorf("planID 解析失败")
    }
    if n == 0 || n > math.MaxInt64 {
        return "", fmt.Errorf("planID 超出有效范围")
    }
    
    return strconv.FormatUint(n, 10), nil
}
```

## 4. 审计结论

### 4.1 总体评价

配置比对模块的核心比对逻辑实现正确，无向边匹配算法设计合理，测试覆盖了主要场景。但在安全性和性能方面存在明显不足，需要进行针对性改进。

### 4.2 优先级建议

| 优先级 | 问题编号 | 修复建议 |
|-------|---------|---------|
| P0 (立即修复) | SEC-001, SEC-002 | 路径遍历和文件上传验证是高危安全问题，必须立即修复 |
| P1 (短期修复) | CON-001, ERR-001, ERR-002 | 并发控制和错误处理影响系统稳定性和用户体验 |
| P2 (中期优化) | PERF-001, PERF-002 | 性能优化可提升用户体验，但不影响功能正确性 |
| P3 (长期改进) | LOGIC-001 | 低危问题，可在后续版本中改进 |

### 4.3 其他建议

1. **增加单元测试**: 针对安全验证逻辑增加边界测试用例
2. **添加审计日志**: 记录文件导入和比对操作的详细信息
3. **实现文件清理**: 定期清理过期的规划文件和报告
4. **添加速率限制**: 对API调用进行速率限制，防止滥用

# NetWeaverGo MIB 库管理演进与规划设计文档

## 1. 概述与愿景
MIB (Management Information Base) 是 SNMP 网络管理的“字典”和“灵魂”。优秀的 MIB 管理模块不仅能解决 OID 到可读文本的翻译，更是支撑设备发现、性能采集、告警解析和配置下发的基石。

**愿景**：将 NetWeaverGo 的 MIB 模块从“简单的文件解析器”演进为“智能的网络设备能力字典”，实现对异构设备的无缝纳管，降低用户的运维门槛。

---

## 2. 核心挑战与现状分析

### 2.1 当前阶段 (V1.0 - 基础解析期)
*   **状态**：支持单个 MIB 文件的上传与解析，能够提取 OID 节点存入数据库。
*   **痛点**：
    1.  **依赖断裂**：单一文件解析容易因缺失父节点或基础依赖（如 `SNMPv2-TC`）导致树结构断裂（“0 节点”问题）。
    2.  **管理碎片化**：用户看到的仅仅是孤立的文件列表，无法对系统拥有的全局监控能力建立直观认知。
    3.  **厂商隔离差**：缺乏全局命名的整合与版本冲突处理机制。

---

## 3. 演进路径与具体规划

我们将 MIB 库管理的演进分为三个阶段（里程碑）：

### 📌 Phase 1：坚实底座 —— 解决“导不进、报错多” (短期规划)
**目标：提升 MIB 导入的成功率和系统的健壮性。**

1.  **内置标准基础 MIB 库 (Built-in Core MIBs)**
    *   **动作**：在系统初始化时，默认预置并隐藏加载所有的 IETF 核心 RFC 标准库（如 `SNMPv2-SMI`, `SNMPv2-TC`, `IF-MIB`, `HOST-RESOURCES-MIB` 等）。
    *   **价值**：用户在导入厂商私有 MIB 时，不再需要到处寻找标准依赖，解决 80% 的导入解析失败问题。
2.  **智能批量导入与依赖拓扑排序**
    *   **动作**：支持整个文件夹或 `.zip` 压缩包的上传。后端自动扫描包内所有文件，提取 `IMPORTS` 语法，构建依赖有向无环图 (DAG)，按拓扑排序自动决定先后导入顺序。
3.  **模块级 (Module) 版本冲突管理**
    *   **动作**：从“文件级管理”升级为“模块级管理”。当导入同名 Module (如 `HUAWEI-SSH-MIB`) 时，对比 `LAST-UPDATED` 字段，实现新版本平滑覆盖旧版本，并给出变更 Diff 提示。

### 📌 Phase 2：字典与实例双重视角 —— 解决“看不懂、不会用” (中期规划)
**目标：建立直观的全局 OID 树和基于设备的 MIB 能力映射。**

1.  **全局 MIB 字典树视图 (Global MIB Dictionary)**
    *   **动作**：在前端开发一个类似于 IDE 文件树的组件，以 `iso(1).org(3)...` 为根节点，将数据库中所有零散的 MIB 模块按 `ParentOID` 缝合在一起。
    *   **价值**：不同厂商的 MIB (如思科的 `.9`，华为的 `.2011`) 挂载在同一个 `enterprises` 分支下，互不干扰，形成 NetWeaverGo 的全局知识图谱。
2.  **设备能力动态发现 (Device Profile Discovery)**
    *   **动作**：增加“一键扫描设备支持 MIB” 功能。后台向设备发送 `snmpwalk` 遍历，将获取到的真实 OID 流与全局字典树进行匹配对齐。
    *   **价值**：生成并展示《单设备真实支持树》，置灰不支持的分支。用户可以清楚地知道这台设备“到底能监控什么”。
3.  **自动分类与标签提取**
    *   **动作**：基于 MIB 节点类型 (`scalar`, `table`, `notification`) 自动为 OID 打标签。例如，自动提取所有 `table` 生成性能监控模板的候选列表。

### 📌 Phase 3：自动化应用赋能 —— 解决“用不好、门槛高” (长期规划)
**目标：让 MIB 数据直接驱动监控告警和自动化运维。**

1.  **SNMP Trap 智能告警翻译**
    *   **动作**：利用 MIB 中的 `NOTIFICATION-TYPE` 节点数据，当接收到干瘪的 SNMP Trap OID 时，自动将其翻译为带有严重级别、故障描述和字段含义的“人话”告警信息。
2.  **基于 MIB 的表单自动生成 (Dynamic UI Generation)**
    *   **动作**：利用 MIB 中的 `MAX-ACCESS (read-write/read-create)` 以及 `SYNTAX` (数据类型/枚举值) 属性。
    *   **价值**：当用户想要下发配置时，前端根据 MIB 自动渲染出下拉框、输入校验等表单，无需手动输入 OID 和十六进制数据。
3.  **云端 MIB 仓库一键同步**
    *   **动作**：构建 NetWeaverGo 的在线 MIB 库生态，支持从云端仓库一键搜索并下载常见厂商型号的 MIB 包，彻底免除手动找包的烦恼。

## 4. 数据库设计优化方向

为了支撑上述演进，建议对当前的数据库模型进行如下调整优化：

*   **分离 MIB 模块与文件介质**：`Module` 实体应独立于 `File`，一个文件可能对应多个 Module。
*   **增加 Device_MIB_Relation 关联表**：用于记录 `snmpwalk` 探测后的结果，标识某设备 `DeviceID` 实际支持的 `ModuleID`。
*   **增强节点的元数据存储**：针对 `ENUM` 枚举值和 `Trap` 告警变量定义（如 `VARIABLES` 列表），单独建立附表进行结构化存储，为后续自动翻译告警做准备。

# 统一执行框架分阶段修复实施方案

基于审计报告：
[unified_execution_audit_2026-03-25.md](./unified_execution_audit_2026-03-25.md)

制定时间：2026-03-25

## 1. 方案目标

本方案用于把当前“任务组主入口已切换，但系统仍存在多套执行体系并行”的状态，逐步收敛为“统一执行框架成为唯一执行真相源”的状态。

本方案要求：

- 每个阶段可独立实施
- 每个阶段有明确输入、输出、验收标准
- 阶段之间尽量单向依赖，避免交叉返工
- 在阶段完成后，系统始终保持可运行

## 2. 当前问题归纳

当前主要问题不是 `taskexec` 不可用，而是它还没有成为全系统唯一执行内核。

需要收敛的对象有 4 类：

- 执行内核：`taskexec`、`executionManager + engine.Engine`、`discovery.Runner + TopologyService`
- 服务暴露层：Wails 当前仍暴露旧服务面
- 前端运行态：仍依赖旧 `execution:snapshot`
- 下游消费面：历史记录、拓扑图谱、规划比对仍基于旧任务模型

## 3. 分阶段原则

### 3.1 先统一内核，再统一展示

如果先改页面、不改后端共享运行时，前端会继续围绕旧事件工作，后续必然返工。

### 3.2 先建桥，再迁移消费方

必须先让统一运行时能稳定对外提供：

- 运行列表
- 当前快照
- 事件推送
- 取消控制
- 历史摘要

否则下游模块无法切换。

### 3.3 保留阶段性兼容层，但逐阶段收窄

每个阶段允许保留兼容代码，但必须明确：

- 哪些是新主路径
- 哪些只是兼容壳
- 哪些将在下一阶段删除

## 4. 分阶段实施方案

## 阶段 1：统一运行时服务化

### 4.1 目标

把 `taskexec` 从“适配器内私有实例”提升为“应用级共享执行服务”，成为后续所有模块迁移的基础。

### 4.2 解决的问题

- `TaskExecutionAdapter` 每次构造都会新建 `TaskExecutionService`
- `taskexec` 没有正式 Wails 服务面
- 新运行时不是全局唯一实例

### 4.3 实施范围

后端：

- `cmd/netweaver/main.go`
- `internal/ui/task_execution_adapter.go`
- 新增或重构统一执行服务，例如：
  - `internal/ui/task_execution_service.go`
  - 或等价的共享运行时服务层

### 4.4 主要改造项

1. 在应用启动阶段创建唯一 `TaskExecutionService`
2. 通过依赖注入方式传入 `TaskGroupServiceV2`
3. 为统一运行时提供稳定的服务接口：
   - `StartTask`
   - `GetTaskSnapshot`
   - `ListRunningTasks`
   - `CancelTask`
   - `ListRuns`
4. 明确旧 `TaskExecutionAdapter` 的职责边界，避免再次私建 runtime

### 4.5 阶段产出

- 应用内唯一 `taskexec` 实例
- 新统一执行服务对外服务面
- `TaskGroupServiceV2` 改为使用共享 runtime

### 4.6 验收标准

- 启动多个任务时，全部运行数据来自同一 runtime 实例
- 重复打开页面或重新创建 service 时，不再出现新的私有 `taskexec` 实例
- `TaskGroupServiceV2` 不再通过“临时构造 adapter”创建 runtime

### 4.7 风险与回滚

风险：

- 服务初始化顺序不当导致空指针或重复启动

回滚方式：

- 保留旧 adapter 包装，但临时退回单服务注册模式

## 阶段 2：统一运行时前端桥接

### 5.1 目标

让 `taskexec` 成为前端可消费的正式状态源，替代旧 `execution:snapshot` 在任务组执行场景中的职责。

### 5.2 解决的问题

- 后端没有把 `taskexec` 快照/事件桥接到 Wails
- 前端无法实时感知统一运行时状态
- `useTaskExecution.ts` 只有占位监听，没有真实数据源

### 5.3 实施范围

后端：

- 统一执行服务
- Wails 事件桥

前端：

- `frontend/src/composables/useTaskExecution.ts`
- `frontend/src/services/api.ts`
- 可能新增 `taskexecStore`

### 5.4 主要改造项

1. 为统一运行时定义前端事件协议，例如：
   - `task:snapshot`
   - `task:finished`
   - `task:event`
2. 后端在 run/stage/unit 状态变化时向 Wails 转发
3. 前端新增专用状态容器，直接消费统一运行时
4. 前端支持通过 `runID` 拉取快照和运行列表

### 5.5 阶段产出

- 稳定的 `taskexec -> Wails -> Frontend` 数据通道
- 前端统一运行时状态管理模块
- 明确的新事件名和类型定义

### 5.6 验收标准

- 启动任务组后，前端无需依赖旧 `execution:snapshot` 即可看到运行进度
- stage/unit 变化能实时反映到页面
- 页面刷新后可通过 `runID` 恢复任务运行态

### 5.7 风险与回滚

风险：

- 事件频率过高导致前端卡顿

回滚方式：

- 保留按需轮询快照作为兜底，事件驱动失败时不影响页面可用性

## 阶段 3：任务执行页和取消链路切换

### 6.1 目标

让“任务执行”页面彻底切换到统一运行时，不再依赖旧 `engineStore` 作为任务组执行主状态源。

### 6.2 解决的问题

- `TaskExecution.vue` 仍以旧 `engineStore.executionSnapshot` 为主
- 停止按钮仍走 `EngineAPI.stopEngine()`
- 拓扑任务有特殊完成提示和跳转捷径，不是统一运行态驱动

### 6.3 实施范围

前端：

- `frontend/src/views/TaskExecution.vue`
- `frontend/src/stores/engineStore.ts`
- `frontend/src/composables/useTaskExecution.ts`
- `frontend/src/App.vue`

后端：

- `TaskGroupServiceV2.CancelTask`
- 统一执行服务查询接口

### 6.4 主要改造项

1. `TaskExecution.vue` 改为以 `runID` 为主键驱动页面
2. 启动任务组后保存返回的 `runID`
3. 停止任务改为调用统一运行时取消接口
4. 拓扑任务和普通任务统一走“运行中 -> 完成 -> 查看结果”的状态流
5. 旧 `engineStore` 仅保留手工执行/备份用途，不再主导任务组执行页

### 6.5 阶段产出

- 任务执行页真正切到统一运行时
- 停止控制正确命中 `CancelTask(runID)`
- 任务组普通任务 / 拓扑任务共享统一执行态页面

### 6.6 验收标准

- 任务组普通任务执行时，页面仅依赖统一运行时快照
- 任务组拓扑任务执行时，不再“启动即提示完成”
- 停止按钮能终止对应 `runID`
- 页面关闭再打开时，仍可恢复当前 run 状态

### 6.7 风险与回滚

风险：

- 任务页逻辑和手工执行逻辑仍共用一部分 UI 状态，切换时容易互相污染

回滚方式：

- 在过渡期对任务组执行态和手工执行态显式分仓

## 阶段 4：拓扑数据消费面统一

### 7.1 目标

把拓扑图谱和规划比对从“发现任务 `taskID` 模型”迁移到“统一运行时 `runID` 模型”。

### 7.2 解决的问题

- `Topology.vue` 仍选择旧 `DiscoveryTaskView`
- `PlanCompare.vue` 仍选择旧发现任务
- 统一运行时执行的拓扑任务，仍通过旧发现任务视图消费结果

### 7.3 实施范围

前端：

- `frontend/src/views/Topology.vue`
- `frontend/src/views/PlanCompare.vue`
- `frontend/src/services/api.ts`

后端：

- 统一执行服务 run 列表接口
- 拓扑查询服务接口

### 7.4 主要改造项

1. 提供“可查询拓扑结果的 run 列表”
2. 拓扑页改为展示统一运行时拓扑 run
3. 规划比对页改为选择统一运行时拓扑 run
4. 页面文案从“发现任务”统一改为“拓扑运行”或等价术语

### 7.5 阶段产出

- 拓扑图谱页使用 `runID`
- 规划比对使用 `runID`
- 旧 `DiscoveryTaskView` 不再作为前端主选择模型

### 7.6 验收标准

- 从任务执行页完成一个拓扑 run 后，可直接在拓扑页按该 run 查看结果
- 规划比对可直接选择统一运行时拓扑 run
- 页面不再依赖 `DiscoveryAPI.listDiscoveryTasks()` 作为主数据源

### 7.7 风险与回滚

风险：

- 旧拓扑查询层仍要求 `taskID`，中间映射设计不清会产生双主键混乱

回滚方式：

- 过渡期保留 `runID -> legacy task_id` 只读映射，但前端只展示 `runID`

## 阶段 5：执行历史统一

### 8.1 目标

把执行历史从旧 `ExecutionRecord` 聚合模型，迁移为统一运行时 run 历史模型。

### 8.2 解决的问题

- 历史页仍按 `RunnerSource` 分类
- `taskexec.ListRuns()` 未被历史页使用
- 新旧执行来源造成统计口径分裂

### 8.3 实施范围

后端：

- `internal/ui/execution_history_service.go`
- 统一执行服务历史查询接口

前端：

- 历史记录页面和历史筛选项

### 8.4 主要改造项

1. 为 `taskexec` 定义历史摘要 DTO
2. 历史页优先展示统一运行时 runs
3. 明确旧 `ExecutionRecord` 的兼容策略：
   - 保留只读
   - 或迁移后逐步废弃
4. 收敛来源字段，避免继续用 `task_group / discovery_service / engine_service / backup_service` 分裂解释主历史

### 8.5 阶段产出

- 基于 run 的统一历史视图
- 历史页筛选项收敛
- 新旧历史模型兼容策略说明

### 8.6 验收标准

- 新框架执行的任务都能在统一历史页查询到
- 历史详情可显示 run/stage/unit 关键信息
- 历史页不再把 `RunnerSource` 作为主分类维度

### 8.7 风险与回滚

风险：

- 老数据无法一次性迁移，导致历史页出现分层展示

回滚方式：

- 先做双视图：统一运行时历史为主，旧记录为兼容分区

## 阶段 6：旧执行框架降级与清理

### 9.1 目标

把旧执行框架从“业务主通道”降级为“兼容壳”或完全移除。

### 9.2 解决的问题

- `TaskGroupService` 内保留完整旧执行实现
- `DiscoveryService` / `TopologyService` 仍兼具执行与查询职责
- `executionManager` 仍在多个页面中承担主状态职责

### 9.3 实施范围

后端：

- `internal/ui/task_group_service.go`
- `internal/ui/discovery_service.go`
- `internal/ui/topology_service.go`
- `internal/ui/execution_manager.go`
- `internal/ui/engine_service.go`

前端：

- 旧事件监听
- 旧执行 API 消费点

### 9.4 主要改造项

1. 删除 `TaskGroupService` 中旧执行主逻辑
2. 重新定义 `DiscoveryService` / `TopologyService`：
   - 保留查询壳
   - 或并入统一执行服务
3. 将 `executionManager` 限缩到手工执行/备份，或继续推进统一替换
4. 清理旧事件名和旧 API 文档

### 9.5 阶段产出

- 明确的新旧边界
- 已删除或降级的旧执行实现
- 文档同步更新

### 9.6 验收标准

- 任务组、拓扑运行、历史查询都不再依赖旧执行实现
- 旧执行逻辑不再出现在主路径
- 新同事仅通过代码结构即可判断统一运行时是唯一主框架

### 9.7 风险与回滚

风险：

- 清理过早导致仍依赖旧模型的边缘功能失效

回滚方式：

- 先打兼容标记和废弃注释，再延后删除

## 5. 阶段依赖关系

推荐执行顺序如下：

1. 阶段 1：统一运行时服务化
2. 阶段 2：统一运行时前端桥接
3. 阶段 3：任务执行页和取消链路切换
4. 阶段 4：拓扑数据消费面统一
5. 阶段 5：执行历史统一
6. 阶段 6：旧执行框架降级与清理

依赖说明：

- 阶段 1 是基础，没有它，后续所有阶段都会继续围绕临时 runtime 工作。
- 阶段 2 是前端迁移的前置条件。
- 阶段 3 可在阶段 2 完成后独立落地。
- 阶段 4、阶段 5 在技术上可并行，但建议先做阶段 4，再做阶段 5，因为拓扑消费模型会影响历史摘要设计。
- 阶段 6 必须最后做。

## 6. 每阶段建议交付方式

每个阶段建议单独成一个 PR 或一个完整变更批次，不要跨阶段混提。

推荐交付粒度：

- 阶段 1：只改后端服务初始化与共享 runtime
- 阶段 2：只补事件桥和统一状态接口
- 阶段 3：只切任务执行页和取消链路
- 阶段 4：只切拓扑页和规划比对
- 阶段 5：只切历史页
- 阶段 6：只做清理和降级

这样做的好处是：

- 问题定位清晰
- 回滚边界清晰
- 测试范围明确
- 避免“功能修复 + 架构重构 + 清理删除”混在一次提交里

## 7. 建议测试策略

每阶段至少执行以下验证：

### 阶段 1

- 启动应用
- 启动普通任务组
- 启动拓扑任务组
- 并发启动多个任务组

### 阶段 2

- 前端实时收到 run/stage/unit 更新
- 页面刷新后恢复任务状态
- 断开监听后重新打开页面仍可恢复快照

### 阶段 3

- 普通任务启动、运行、取消、完成
- 拓扑任务启动、运行、取消、完成
- 停止按钮命中正确 run

### 阶段 4

- 拓扑页按 `runID` 查看结果
- 规划比对按 `runID` 选择数据源
- 老 run 与新 run 的查询兼容性

### 阶段 5

- 历史列表加载
- 历史详情加载
- 历史筛选与统计

### 阶段 6

- 全量回归任务组执行
- 拓扑查询回归
- 手工执行/备份回归
- 编译与测试全通过

## 8. 最终交付标准

当以下条件全部满足时，可判定“统一执行框架修复完成”：

- 任务组执行、拓扑运行、历史记录、拓扑图谱、规划比对均围绕 `runID` 工作
- 前端任务执行态只消费统一运行时，不再依赖旧 `execution:snapshot`
- 停止/取消只走统一运行时取消接口
- `taskexec` 成为唯一主执行服务
- 旧任务组执行逻辑不再保留在主路径
- 旧发现任务模型不再作为前端主入口模型
- 历史页不再以 `RunnerSource` 作为主分类模型

## 9. 推荐落地节奏

如果按风险控制优先，建议这样推进：

- 第 1 周：阶段 1 + 阶段 2
- 第 2 周：阶段 3
- 第 3 周：阶段 4
- 第 4 周：阶段 5 + 阶段 6

如果按最小可见收益优先，也可以这样推进：

- 先做阶段 1 + 阶段 2 + 阶段 3
- 先让任务执行页真正统一
- 再处理拓扑和历史下游模块

## 10. 结论

当前最合理的修复策略，不是继续做零散补丁，而是按阶段收敛：

- 第一阶段统一服务实例
- 第二阶段打通前端桥
- 第三阶段切主页面
- 第四、第五阶段迁移下游消费面
- 第六阶段集中清理旧框架

这样可以保证每一步都是独立可验收的，同时避免再次引入“新框架已经接入，但旧框架仍是事实真相源”的混乱状态。

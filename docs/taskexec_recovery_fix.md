# 任务执行状态机异常退出修复说明

## 问题背景

当前任务执行系统存在一个全局性一致性问题：运行实例进入 `pending` / `running` 后，如果应用进程崩溃、被强制结束，或者用户直接关闭程序，内存中的运行时上下文会消失，但数据库中的运行记录不会自动进入终态。

这会导致以下连锁问题：

1. `task_runs` 中仍保留活跃状态。
2. 对应 `task_run_stages` / `task_run_units` 可能仍停留在 `running`。
3. 新任务启动时，目标设备冲突检测仍把这些运行视为活跃运行。
4. 任务组编辑/删除入口会因为“存在活跃运行”而被持续阻塞。
5. 前端列表与执行页会长期显示任务仍在执行，形成“卡死”。

这不是单点函数问题，而是**运行时内存态、数据库持久态、UI 判定逻辑三者之间缺少统一收口机制**导致的全链路状态一致性问题。

---

## 根因分析

### 1. 运行状态写入早于真正执行完成

任务在创建后会先进入 `pending`，执行开始后再写成 `running`。但终态写入依赖正常执行收尾、取消流程或者显式补偿路径。

一旦进程异常退出：

- 内存中的运行时上下文丢失；
- [`RuntimeManager`](../internal/taskexec/runtime.go) 不再持有该运行；
- 但数据库中的 `run/stage/unit` 状态不会自动收敛。

### 2. 原有补偿能力只在“手工取消离线运行”场景生效

原本已经存在补偿取消能力，用于运行实例不在内存中时，对离线运行执行补偿取消。

但该能力没有在应用启动时主动执行，因此：

- 崩溃遗留任务不会被自动修正；
- 系统只能等用户再次点“取消”，才可能走到补偿路径；
- 如果前端没有对这类历史运行暴露取消入口，就会长期卡死。

### 3. 活跃运行判定分散，且部分逻辑依赖内存快照

活跃运行有两条来源：

- 数据库历史运行记录；
- 内存中 `snapshot/runningRuns` 里的在线运行快照。

异常退出后，内存态天然为空，但数据库仍保留 `pending/running`。如果某些业务逻辑只看内存、某些逻辑又直接看数据库，就会出现判定不一致：

- 启动冲突检测把遗留运行视为活跃；
- 任务组编辑入口也把遗留运行视为活跃；
- 但实时执行视图可能已经看不到对应内存快照。

本质上，这是**活跃态语义没有统一抽象**导致的状态分裂。

---

## 本次修复的总体策略

采用最稳妥的全局方案：

> **不新增状态，所有因进程退出而遗留的 `pending` / `running` 运行，在应用启动时统一自动收敛为 `cancelled`。**

这样做的好处：

1. 不引入新的状态枚举和前端适配复杂度。
2. 与现有“用户取消”语义兼容，终态处理简单明确。
3. 可以立即解除设备冲突占用与任务编辑锁定。
4. 保证数据库、运行时、前端展示统一使用同一套终态判断。

---

## 代码层修复点

### 1. 统一活跃态语义

在状态定义中补充统一语义：

- `ActiveRunStatuses()`：返回所有活跃运行状态；
- `IsActiveRunStatus(status string)`：统一判断一个运行是否仍然活跃；
- 同时把 `skipped` 视为 stage 终态，避免阶段终态语义不完整。

这样所有上层模块都不再各自硬编码 `pending/running`，而是统一走状态语义函数。

### 2. 仓库层活跃运行查询统一覆盖 `pending + running`

[`ListRunningRuns`](../internal/taskexec/persistence.go) 原先只查询 `running`，现在统一查询所有活跃运行状态。

这样做的意义是：

- 启动恢复时能覆盖刚创建但还未来得及切到 `running` 的遗留任务；
- 启动冲突检测对“未真正收尾的活跃运行”有一致视角；
- 不再遗漏停留在 `pending` 的半启动运行。

### 3. 新增启动恢复：应用启动时自动补偿取消遗留活跃运行

在任务执行服务启动阶段，先执行一次离线恢复：

1. 查询所有活跃运行；
2. 对每个非终态运行执行补偿取消；
3. 把 `run/stage/unit` 全部统一收敛到 `cancelled`；
4. 写入 `finishedAt`，确保历史记录完整闭环；
5. 记录日志，便于排查应用异常退出后的恢复行为。

这样用户再次打开程序时，遗留的执行中任务会被自动清理，不再需要人工介入。

### 4. 补偿取消逻辑增强

补偿取消逻辑现在具备两个特性：

- 对 `run` 强制写入 `cancelled + finishedAt`；
- 对 `stage/unit` 仅在其仍为非终态时才继续改写，避免覆盖已收口状态；
- 对被改写的 `unit` 补写 `error_message = run cancelled by compensation`。

这保证了补偿逻辑既能完成全局收口，又不会误伤已经进入终态的数据。

### 5. 任务组编辑/删除/活跃计数统一基于终态语义判断

任务组服务原先直接写死：

- `pending` 或 `running` 即不可编辑；
- 活跃数量部分依赖实时内存快照。

现在统一改成：

- 只要 `IsActiveRunStatus(status)` 为真，就视为活跃；
- 活跃计数直接基于仓库中的运行状态统计；
- 不再依赖仅存在于当前进程内存中的在线快照。

这样即便发生异常退出，只要应用下次启动完成恢复，编辑与删除入口就会自动解除锁定。

---

## 修复后的状态流转

### 正常执行路径

`pending -> running -> completed/partial/failed/cancelled/aborted`

### 异常退出后的恢复路径

`pending/running --(应用重启恢复)--> cancelled`

### 恢复收敛范围

- Run：`cancelled`
- Stage：非终态的统一收敛为 `cancelled`
- Unit：非终态的统一收敛为 `cancelled`
- 时间字段：补写 `finishedAt`
- 原因字段：Unit 补写补偿取消原因

---

## 为什么这是“全局修复”而不是局部补丁

因为这次修改同时覆盖了四层：

1. **状态语义层**：统一活跃态/终态判断。
2. **仓库查询层**：统一活跃运行查询口径。
3. **运行时启动层**：新增恢复收口动作。
4. **业务使用层**：编辑锁、冲突检测、活跃计数全部复用统一语义。

这意味着修复并不是在某个按钮、某个页面、某个取消接口上打补丁，而是把“异常退出后如何收口活跃运行”沉入了任务执行架构本身。

## 已补充的验证点

本次已补充并增强以下验证：

1. 补偿取消后，`run/stage/unit` 都会进入 `cancelled`。
2. 补偿取消后，会正确写入 `finishedAt`。
3. 启动恢复会同时处理遗留的 `pending` 与 `running`。
4. 已处于终态的运行不会被启动恢复错误改写。
5. 构建脚本 [`build.bat`](../build.bat) 已通过，说明整体编译链路正常。

## 影响评估

### 正向影响

- 异常退出后的卡死运行会在下次启动时自动解除。
- 任务组可编辑性会自动恢复。
- 新任务发起时不会再被历史僵尸运行长期占用目标设备。
- 历史记录状态更可信，避免“界面显示执行中但实际程序已经不存在”的假象。

### 设计取舍

当前方案选择“统一收敛为 `cancelled`”，而不是新增 `interrupted`：

- 优点是简单、稳定、改动面可控；
- 缺点是无法区分“用户主动关闭程序”和“程序异常崩溃”。

对于当前新建项目阶段，这是最稳妥、最低复杂度、收益最高的方案。

---

## 后续可演进方向

如果后续需要更强的运行恢复能力，可以继续扩展：

1. 引入 `interrupted` 状态，区分异常退出与人工取消。
2. 为运行实例增加心跳/lease 机制，而不是仅依赖启动补偿。
3. 对支持幂等恢复的任务类型提供“继续执行”而不是“统一取消”。
4. 在 UI 历史记录页增加“该运行由系统启动恢复自动取消”的标识。

但在当前阶段，不建议先引入这些复杂度，先把**状态一致性和可运维性**稳定住更重要。

# NetWeaverGo 项目问题解决思路（复核版）

> 基于 `docs/project_analysis_report.md` 与 `docs/crypto_removal_plan.md` 整理  
> 仅提供解决思路，不提供代码  
> 复核时间：2026-03-20

---

## 1. 本次复核结论

结合你恢复的 `docs/crypto_removal_plan.md` 和当前代码，先给出几个关键结论：

1. 历史加密清理大概率已经完成。
   - 当前仓库中未发现 `internal/config/crypto.go`
   - 未发现 `registerCredentialCallbacks`、`devicePasswordContext`、`CredentialCipher` 等残留引用
   - 也未发现 `.credential_key` 文件
2. 这意味着报告里“历史兼容代码待清理”的 H1-H3，现状更接近“已完成”而不是“待做”。
3. 但清理加密后带来的安全暴露仍然存在，而且当前代码里仍有几个高风险点：
   - `internal/ui/device_service.go` 仍输出明文密码调试日志
   - `internal/ui/query_service.go` 和设备列表查询仍会把密码带给前端
   - 前端密码查看能力已被默认放大到“列表/查询即可拿到”，这比“允许查看已有密码”更宽

## 2. 已确认的决策前提

以下内容已按你刚确认的边界来设计：

- 密码安全存储只需要支持 Windows
- SSH `insecure` 模式保留，但仅作为高级兼容开关
- 前端编辑设备时允许查看已有密码
- `crypto_removal_plan.md` 可作为历史清理依据继续参考

这几个前提会直接影响 S1、S2、S4 和 H1-H5 的处理方式。

## 3. 架构问题

### A1. UI 服务层与 Config 层耦合过紧

**思路**

1. 将 `config` 包的职责收敛到“配置管理、路径管理、数据库初始化、运行时常量”。
2. 把设备、任务、命令组、设置等 CRUD 逻辑抽到 `repository` 或 `store` 层。
3. UI Service 只面向接口调用，不直接依赖 `config.LoadXXX` / `config.UpdateXXX`。
4. 需要跨资源编排的流程再加一层 application/usecase service，避免 repository 承担业务编排。
5. 后续测试通过 mock repository 实现，不再直接依赖数据库。

**建议顺序**

1. 先抽设备资产读写
2. 再抽设置与任务读写
3. 最后清理 `config` 包中的业务函数

### A2. Engine 状态管理复杂

**思路**

1. 先补一份状态流转表，明确每个状态的进入条件、退出条件、禁止行为和副作用。
2. 将 `Engine` 的职责拆成几个小块：
   - 状态流转
   - 生命周期上下文
   - 事件转发
   - 关闭流程
3. 对外暴露固定入口，如 `Start`、`Pause`、`Resume`、`Stop`、`Close`，不允许外部绕过状态机直接改内部字段。
4. 将进入 Running、Closing、Closed 等状态时的动作统一放到钩子里，避免逻辑散落。

**实施原则**

- 不建议整体重写引擎
- 先做“状态约束收口”，再做内部结构拆分

### A3. 前端路由未实现懒加载

**思路**

1. 非首屏页面改为动态导入。
2. 按业务域拆分 chunk，例如设备、发现、拓扑、配置生成、设置。
3. 为懒加载页面补 loading 或骨架屏，避免切页空白。
4. 高频页面可预取，但不建议全部预加载。

## 4. 代码冗余问题

### R1. 密码处理逻辑重复

**思路**

1. 抽一套统一的密码合并规则，单设备更新和批量更新都复用。
2. 明确语义：
   - 空值表示“不修改”
   - 新值表示“更新”
   - 如果未来需要“清空密码”，必须增加单独语义，不要继续复用空字符串
3. 统一返回“密码是否变化”的结果，供日志和审计使用。

**联动**

- 这项要和 S4 一起设计，否则前后端语义容易不一致。

### R2. Discovery Runner 重复数据结构

**思路**

1. 将重复匿名结构体抽成具名 DTO。
2. 数据库查询行结构与业务传输结构分开命名，避免混用。
3. 如果只在 discovery 模块内部使用，就放模块内，不必上升到全局模型层。

### R3. SSH 认证逻辑重复

**思路**

1. 统一 SSH 认证方法构造入口。
2. 密码认证、键盘交互认证、后续若加入密钥认证，都走同一组装流程。
3. 主机密钥校验策略与认证方法构造分离，不要放在一个函数里耦合处理。

### R4. 日志调试信息泄露密码

**思路**

1. 立即移除 `internal/ui/device_service.go` 中的明文密码日志。
2. 后续如果要保留调试信息，只记录：
   - 是否为空
   - 是否更新
   - 长度或掩码摘要
3. 将“敏感字段禁止直接打日志”写入代码评审规则。

**优先级**

- 第一批必须处理

### R5. 前端 API 类型定义重复

**思路**

1. 以后端生成 bindings 作为唯一 DTO 来源。
2. `frontend/src/services/api.ts` 可以保留为 API 聚合层，但不再重复定义后端类型。
3. 前端自定义类型只保留表单态和视图态。
4. 把 bindings 生成和前端编译检查串到同一流程里，避免漂移。

## 5. 安全问题

### S1. 密码明文存储

**已确认前提**

- 只支持 Windows

**建议方案**

Windows 场景下，优先按 DPAPI 设计，不建议继续围绕自研文件密钥方案扩展。

**推荐思路**

1. 定义统一的凭据存储接口，隔离具体加解密实现。
2. 数据库中不再保存纯明文密码，而是保存 DPAPI 保护后的密文字段。
3. 仅在以下场景解密：
   - 发起 SSH 连接
   - 用户显式查看/编辑已有密码
4. 对数据库备份、导出、日志都默认按“敏感数据”处理。
5. 因为 `crypto_removal_plan.md` 已明确无需兼容历史加密数据，所以这里不需要处理旧密钥迁移，只需要考虑“当前是否已有明文数据要转成 DPAPI 密文”。

**分阶段建议**

1. 短期止血：
   - 立刻移除密码日志
   - 控制数据库和日志目录权限
   - 列表/查询接口停止回传密码
2. 中期落地：
   - 引入 DPAPI 保护凭据
   - 调整查看密码与执行连接的解密入口
3. 长期增强：
   - 如后续有更高合规要求，再评估数据库整体加密或凭据独立仓储

### S2. SSH 主机密钥校验策略宽松

**已确认前提**

- `insecure` 模式保留为高级兼容开关

**建议方案**

1. 默认策略仍应固定为 `strict`。
2. `insecure` 放到高级设置区，不放在常规配置首屏。
3. 用户切换到 `insecure` 时，要求显式确认，并在 UI 中持续显示风险提示。
4. 每次使用 `insecure` 或 `accept_new` 建连，都写审计日志，至少记录：
   - 设备地址
   - 使用的校验策略
   - 时间
5. `accept_new` 需要真正保存首次接受的主机指纹，并在指纹变化时给出警告。

**建议边界**

- 不建议删除 `insecure`
- 也不建议让它成为默认选项或静默生效选项

### S3. 日志敏感信息泄露

**建议方案**

1. 将脱敏下沉到日志模块统一出口，而不是各业务点自行决定。
2. 建立敏感字段清单，包括：
   - password
   - secret
   - token
   - key
   - 带认证信息的连接串
3. 对普通日志、错误日志、原始设备输出分别定义脱敏策略。
4. 禁止高风险路径直接把结构体或错误原文原样打出。
5. 补回归样例，覆盖网络设备配置中常见敏感片段。

### S4. 前端密码暴露

**已确认前提**

- 前端编辑设备时允许查看已有密码

**处理原则**

允许查看，不等于所有接口默认下发密码。

**建议方案**

1. 设备列表接口、分页查询接口、搜索接口默认不返回密码。
2. 保留一个专门的“查看/编辑凭据”入口，例如：
   - 打开编辑弹窗时按需获取
   - 或在编辑弹窗内用户点击“显示密码”后再获取
3. 密码查看能力只保留在单设备级别，不扩散到列表、批量查询、批量导出。
4. 前端拿到密码后只在本地临时状态中保存，关闭弹窗即清空，不进入全局 store。
5. 如后续上 DPAPI，这个入口就是唯一解密出口之一。

**当前代码下的直接问题**

- `ListDevices()` 与 `QueryService` 当前都会把密码带给前端，这超出了“允许查看已有密码”的最小范围，建议收紧。

## 6. 性能优化问题

### P1. 数据库连接池配置

**思路**

1. 先观测 SQLite 写锁冲突、慢查询、连接等待时间。
2. 连接池参数统一收口，不要分散硬编码。
3. 对 SQLite 来说，重点是减少写竞争，不是盲目增加连接数。

### P2. 事件总线缓冲区大小

**思路**

1. 区分关键事件和高频进度事件。
2. 关键事件必达，进度事件允许合并或丢弃旧值。
3. 增加队列长度、丢弃次数、消费延迟统计，再决定是否调整容量。

### P3. 并发数硬编码

**思路**

1. 引擎、发现、备份等模块统一走同一套并发配置解析规则。
2. 默认值、最大值和用户自定义值分层管理。
3. UI 要明确提示“并发数不是越大越好”。

### P4. 路由懒加载

**思路**

- 与 A3 合并实施即可，不必拆成两个独立改造任务。

### P5. 大列表虚拟滚动

**思路**

1. 先确认真实设备规模和性能瓶颈。
2. 如果设备列表长期超过百级或数百级，再引入虚拟滚动。
3. 若表格交互复杂，也可以先用分页，避免虚拟滚动影响多选、批量编辑和行内操作。

### P6. 状态订阅优化

**思路**

1. 将执行快照拆成高频状态和低频详情。
2. 大对象优先用浅层响应与整对象替换。
3. `computed` 只保留轻量派生值，重计算逻辑改为显式更新。

### H4. `internal/ui/device_service.go` 密码处理逻辑

**复核结果**

- `GetDeviceByID()` 仍会输出密码日志

### H5. `internal/ui/query_service.go` 密码处理逻辑

**复核结果**

**结论**

- 从“去除历史脱敏循环”的角度看，这项已完成
- 但从安全边界看，现在暴露范围过大，应按 S4 收紧为“按需查看”

**整体判断**

`crypto_removal_plan.md` 中的“移除旧加密链路”基本已反映到当前代码；后续重点不再是继续删旧代码，而是基于新决策重建一套更可控的 Windows 凭据保护与前端暴露边界。

## 8. 待验证项补充思路

### V1. 路径遍历防护

**思路**

1. 统一所有文件路径输入校验。
2. 对导入导出路径、数据库镜像路径、存储根目录拼接结果做规范化检查。
3. 限制关键文件读写只能落在允许目录内。

### V2. 输入验证不足

**思路**

1. IP、端口、协议、用户名、分组名建立统一校验规则。
2. 前后端都校验，但以后端为准。
3. 批量导入和批量编辑要能返回逐条错误，方便定位。

## 9. 推荐实施顺序

- R4 删除密码日志

- S4 收紧列表/查询接口的密码返回

- R1 统一密码合并规则

### 第三批

- S1 Windows DPAPI 凭据保护
- A2 Engine 生命周期治理
- P2 事件背压与事件分类
- P5 大列表分页或虚拟滚动

### 可关闭项

- H1-H3 可视为已完成
- H4-H5 不再作为“历史清理任务”推进，而并入 S3/S4 安全整改

## 10. 这次复核后的最终建议

如果只看你现在已经确认的产品边界，最合适的方向是：

1. 不回退到旧的自研加密链路。
2. 先把“密码到处都能拿到”的暴露面收紧。
3. 然后按 Windows-only 路线引入 DPAPI。
4. `insecure` 模式保留，但必须显式警告和审计。
5. “允许查看已有密码”只保留在单设备显式操作中，不保留在列表和查询默认返回中。

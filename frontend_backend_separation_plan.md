# NetWeaverGo 内核后置架构升维计划书（含 Wails 保留方案）

## 1. 架构目标澄清
当前的 `NetWeaverGo` 虽然是基于 **Go + Vue (Wails v3)** 搭建的跨平台桌面端应用，但在目前的开发进程中存在一个不合理的反模式：**大量的核心业务逻辑漏散和驻留在了前端 Vue 的组件中**。

**核心改版受命：**  
我们**不抛弃** Wails 的打包方式及桌面客户端（无需搭建纯 Web 甚至提供 HTTP Axios ）。但要求：
- 前端 (Vue) 退化为**纯正的“画皮”**，即：**仅接收组装好的最终形态数据并渲染视图**。
- 后端 (Go) 提升为**绝对的“大脑”**，所有的排序、过滤、状态机维护、大批量的日志归纳、以及复杂配置组合的生成计算，全部交由 Go 完成。此时 Wails Binding 会被用作纯粹的数据流通管道，而非简单数据库 CRUD 的暴漏。

## 2. 现状病理与开刀处（核心重构区）

### 2.1 任务执行与状态归纳层（重灾区）
- **当前前端实现 (`TaskExecution.vue`)**：
  目前前端接收了细碎散装的 `EngineEvent` (如单行日志接收、设备开始等)。前端在内部维护了一个巨大的响应式数组 `execDevices`，并自己提供 `batchUpdateDevices` 和 `addLog` 以及**日志条数截断限流机制**（`LOG_CONFIG` 里的上限截断与警告判定）。
- **重构方案 (Go 接管)**：
  前端只负责调用 Go：`GetDataSnapshot()`。
  后端 Go 的 `collector.go` 应该不仅是一个进度终端大盘打印器，而是必须在内存中构建一棵类似于 Redux Store 的**实时全量状态树 (State Tree)**，其中包含了所有正在运行的设备的健康度、已截断好的日志集和聚合进度条百份比计算结果。前端只需基于 `time.Ticker` 触发 Binding (或 Wails 的 Event 下发) 去取这棵成品状态树对象，即可**原样绑定到 UI 渲染而勿需经过任何计算（完全砍掉前端日志过滤截断运算代码）**。 

### 2.2 配置/命令工厂装配组（ConfigForge）
- **当前前端实现 (`ConfigForge.vue`)**：
  该页面通过用户选择的各种变量拼接宏，组合网络基础指令，同时该页面的数据拼装、切分和参数变量关联是在 JS 中使用各种正则提取器甚至数组 Map 计算出来的。
- **重构方案 (Go 接管)**：
  前端提交**原始表单要求 (Raw Demands)**。
  增加后端的 `ConfigBuilderService`。所有的正则变量插值（例如 `$ip`, `$gateway` 的生成规则）与对已有参数校验，要在 Go 中实现，在生成后只将可直接打印或者可一键发配最终态的数据列表响应给前端展示即可。

### 2.3 数据过滤与排序能力后撤
- **当前前端实现 (`Devices.vue` / `Tasks.vue`)**：
  获取列表后，诸如对特定标签的搜索过滤、基于执行状态的本地 `computed` 实时排序筛选全都堆积在 Vue 的逻辑层 (`filteredTasks`, 分页等)。
- **重构方案 (Go 接管)**：
  增加后端的 `Listxxx(QueryOptions)` 提供带条件查询的函数。  
  前端遇到搜索、过滤请求时，直接传递条件回发给后端的 Wails Binding 函数。由 Go 在服务端利用 `gorm` 进行内存/数据库的 Where 聚合筛选以及排序后返回结果数组给前端替换响应式变量。

## 3. 具体实施阶段路线

### 阶段一：服务模型升级与数据状态树建设（后端为主）
1. 在 `internal/ui` 创建 / 升级统一对外的 **View Model (视图模型层)**：定义一系列 `XxxViewState` 结构体，其字段严丝合缝对齐前端所需要直接使用的数据。
2. 重写 `ProgressTracker` （`collector.go`），为任务系统植入 `GetExecutionSnapshot()`。进度统计、状态判断（是成功还是警告）及文本过滤拼接都在这个快照内实现完备。
3. 扩展当前各个 Wails Service 的方法，确保接受查询条件如 `SearchQuery string`, `FilterStatus string`。

### 阶段二：前端无脑化降级裁剪（前端为主）
1. 深入 `TaskExecution.vue`，干掉所有的 JS 对象状态合并、状态字典、事件聚合逻辑及限制上限和日志着色的切分器。
2. 更改 `useEngineEvents` Composable 的行为机理，由零散拼接变为：“每次收到需要更新的信号时，调用后端的快照取回一次成品的视图 JSON 并赋值赋”。
3. 深入各 CRUD 增删改查页面（如 Device, TaskGroup），移除非必须的 `computed` 过滤，挂载到底层的 `API.listXxx` 参数请求透传。

### 阶段三：深层业务工厂后移
1. 建立后端核心：`internal/forge` 开发配置生成器，取代前端的配置拼装能力。
2. 将 `ConfigForge.vue` 原有的本地组合算法全部作废，替换为统一的 Go 处理。

## 4. 实施总结与效益
- **业务安全与稳定性**：Go 语言作为强类型与高性能标杆，比浏览器 JS 引擎更擅长处理大型正则匹配及大量字符串日志拼装切片，极大防止了海量网格设备执行命令时引发的客户端内存泄漏及浏览器卡顿。
- **代码可维护性升维**：由于 Wails 这座桥接被限制到了仅流通已完成结算加工后的**最终成相数据（View Model）**，开发工作职能彻底分家，且让重核调度代码被牢牢封装在底层。
- **改造成本**：在保留 Wails 框架下迁移此项，成本非常低。不需要额外部署或折腾 Nginx 及 Vite HMR 跨域等琐事，随时可以启动第一阶段开刀。
